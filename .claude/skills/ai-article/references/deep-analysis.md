# 深度拆解类参考

本文件是「深度拆解类」风格的写法指南，基于小林coding的 Skill 拆解文章提炼。

## 核心特征

深度拆解类文章的竞争力不是文笔，是**调查深度**。读者来看这类文章，是因为作者做了他们没时间做的事：读源码、翻 PR、跑实验、拆机制。

## 结构模板：问题链

整篇文章围绕一条问题链展开，每个章节回答一个问题，问题之间有递进关系。

```
开头（3段内）：冲突 → 结论 → 收益
  - 冲突：大多数人对X的理解是错的 / 你是不是也遇到了Y问题
  - 结论：一句话说清楚本文最大的发现
  - 收益：读完你会知道什么

## 01、第一个问题（纠正误解/建立正确认知）
## 02、第二个问题（分类/全景图）
## 03、第三个问题（机制/原理，本文最硬核的部分）
## 04、第四个问题（最佳实践）
## 05-07、递进问题（高阶用法/团队/度量）

## ending
  - 浓缩成 2-3 句话
  - 给出最值得立刻做的一件事
```

## 写法示范（摘自小林文章）

### 示范1：悬念开场

```
先抛个问题：如果让你现在描述一下什么是 skill，你会怎么说？

我猜不少人的答案是：「就是一个写了操作步骤的 markdown 文件，Claude 需要的时候会去读。」

Anthropic 在博客里点名说了，这是他们听到的关于 skill 最常见的误解。
```

要点：先让读者说出自己的答案，再告诉他答案是错的。制造认知落差。

### 示范2：源码证据 + 解读

```
讲完原理，我去源码里取了证，结果发现真实情况比博客说的还要苛刻。

skill 清单注入 context 是有预算的，而且预算紧得吓人：

export const SKILL_BUDGET_CONTEXT_PERCENT = 0.01
export const MAX_LISTING_DESC_CHARS = 250

这段代码在 src/tools/SkillTool/prompt.ts，两个常量翻译成人话：整张 skill 清单只允许占用 context 窗口的 1%；单个 skill 在清单里的描述最多 250 个字符。

250 个字符之后会发生什么？源码里写得明明白白：

return desc.length > MAX_LISTING_DESC_CHARS
  ? desc.slice(0, MAX_LISTING_DESC_CHARS - 1) + '…'
  : desc

超出 250 字符的部分直接被砍掉，换成一个省略号。
```

要点：
- "我去源码里取了证"——明确告诉读者你做了什么
- 先贴代码，再"翻译成人话"
- 标注代码来源路径
- 解读时用具体场景："你在第 300 个字符处写的精妙触发条件，模型从头到尾就没见过"

### 示范3：章节收尾的实操建议（加粗独立成段）

```
description 的前 250 个字符，决定了你的 skill 是工具还是摆设。
```

```
skill 不是收藏品，贵精不贵多。
```

```
先别急着写一堆 skill，把「让 Claude 自己验证工作成果」这一件事做好，回报最大。
```

要点：每个章节结尾一句话总结，既是结论也是行动指南。读者扫读时只看这些加粗句子就能抓住全文骨架。

### 示范4：类比让抽象概念可感知

```
markdown 文件版的 skill，相当于你给新同事发了一条微信：「部署流程是先这样再那样」。而文件夹版的 skill，相当于你给他一个工位：桌上有操作手册，抽屉里有现成的工具，墙上还贴着前任留下的「这台打印机会卡纸，要先按两下」的便利贴。
```

要点：类比要具体到画面感，不能是"A 就像 B"这种空洞比喻。要让读者在脑子里能"看到"那个场景。

### 示范5：章节之间的自然过渡

```
触发问题解决了，Claude 终于肯打开你的 skill 了。下一个问题：正文该写什么？
```

```
明白了这个机制，「为什么不触发」的答案就浮出来了。
```

要点：前一个问题的结论自然引出后一个问题，读者有"顺流而下"的阅读感。

### 示范6：流程逻辑型证据

```
整个 skill 的触发判断只有三步：

1. 收集所有 skill 的 name + description，拼成一张清单注入 context
2. 模型每轮对话时扫一眼清单，判断当前任务是否匹配某个 skill
3. 匹配上了才发起调用，这时 SKILL.md 全文才被加载进对话

注意第 1 步：它只注入 description，不注入正文。
这意味着模型做匹配决策时，唯一的依据就是那一行 description。
正文写得再好，description 没写对，模型从头到尾就不知道你存在。
```

要点：
- 把代码逻辑翻译成了一个编号步骤的流程
- 点明流程中对读者有实操意义的节点（"只注入 description"）
- 结尾指向行动："description 没写对，模型从头到尾就不知道你存在"

## 深层调查操作方法

1. clone 或本地打开目标项目源码
2. 根据文章主题定位关键文件（grep 关键词、看目录结构）
3. 找到支撑观点的具体代码片段（常量定义、核心逻辑、注释）
4. 在文章中先展示代码证据，再给出自己的解读——让读者同时看到"事实"和"判断"
5. 筛选标准：只保留满足以下任一条件的代码片段——
   - 它推翻了一个常见误解（"大家以为是 X，源码告诉你是 Y"）
   - 它量化了一个模糊概念（"不是'有限制'，而是'限制为 250 字符'"）
   - 它暴露了一个设计取舍（"为了 A 牺牲了 B"）
   - 不满足的代码片段再有趣也不贴，它会稀释文章的信息密度

## 写作硬规范

把上面的示范提炼成可执行的硬约束：

1. **内容来源（一手证据）**：每个核心观点必须有一手证据支撑——源码片段（标注文件路径和关键行）、官方博客/文档原文、实测数据（自己跑出来的结果）、GitHub issue/PR 讨论原文。禁止"据说""业内普遍认为""有人发现"这类无来源表述支撑核心论点。
2. **正文节奏（章节级四拍）**：每个章节遵循"**悬念 → 证据 → 解读 → 实操建议**"——用具体问题或反直觉现象开场，展示源码/数据/原文让读者看到事实，用自己的话点明"这对你意味着什么"，最后一句话给出可执行建议（加粗或独立成段，方便扫读）。
3. **代码片段规范**：只截取和观点直接相关的片段（5-15 行），标注来源（文件路径或仓库名 + 相对路径）；代码前一句说"在做什么"，代码后一句说"意味着什么"，没有解读的代码就是噪音。
4. **章节衔接**：用前一个问题的结论自然引出下一个问题（"这就引出了下一个问题""明白了这个机制，X 的答案就浮出来了"），让读者感受到问题链的递进，而不是孤立知识点堆砌。

## 常见失败模式

**堆砌源码不解读**：贴了 200 行代码，前面一句"我们来看源码"，后面一句"可以看到实现了 X 功能"。读者看完依然不知道这代码和自己有什么关系。问题在于：没有回答"so what"。

**用源码装饰结论**：先有结论再找代码来"证明"，而不是从代码中发现结论。读者能感觉到——你即使不贴代码，结论也不会变。这种源码是装饰品，不是证据。

**平铺知识点无递进**：7 个章节各自独立，像百科词条的合集。读者每读完一节都可以关掉页面，说明问题链没有建立起来。好的递进是：读完第 3 节你不得不想知道第 4 节。

## 禁忌

1. **禁止只有结论没有证据**：每个核心观点必须有对应的源码/数据/原文支撑
2. **禁止罗列知识点**：章节之间必须有递进关系，不是并列的知识卡片
3. **禁止堆砌代码**：每段代码都要有"为什么放在这"的理由，没有解读的代码就是噪音
4. **禁止"据说"式引用**：来源不明的信息宁可不写，写了就标来源
5. **禁止全篇同一节奏**：有的章节重证据（贴代码），有的章节重类比（讲故事），有的章节重数据（列表格），节奏要有变化
