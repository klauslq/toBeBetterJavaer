# 面试对话类专属规范

> 仅在步骤 4 选择「面试对话类」时读取本文件。高赞正例全文见 `anshui-yin-mianshi.md`；答题手册的逐题定稿正例见 `agent-mianshi-tengxun-2.md`（2026-07-22 与二哥逐题校准 7 轮的成果，写作前对照）。

## 标题（面试类）

素材里有现成标题直接用，末尾加"（附Agent面试题）"式括号后缀标注干货属性。公众号标题上限 64 字，超了优先砍薪资细节数字，保住冲突和后缀。

## 正文结构（覆盖通用规则）

`## content` 包裹所有面试题 → 每题 `### 01、问题`。章节标题是问题的加问号；题目含两问的（"怎么做，怎么判断"），第二问拆成 `####` 四级标题。

**对话体框架**：

- 老王的问句用"动作 + 衔接词 + 引号"起手："老王在他的小本本上打了个勾继续提问。""老王往前倾了倾身子，继续问。"——动作要具体带画面，允许微幽默，后面必须接"继续问"类衔接，动作不悬空。全篇动作 3-5 处，不要每题都有，动作和回答质量呼应
- 短题可以裸引号直接问（"badcase 怎么回流？"），维持快问快答的节奏
- 问句里术语写全："提示词的 system 层""超出上下文窗口"，生客第一次读不卡壳
- 回答默认带""引号；成段的概念讲解可以脱离引号直接叙述（04 定稿），引号留给真正的对话往来
- why 型追问升级成 `####` 四级标题，读者扫标题就能看到干货点，不埋在引号里自问自答

**对话体的边界（强制）**：对话框架是活的，回答内容是严谨的。老王的言行、"我"的临场反应走体验叙事轨；回答里的技术内容走技术论述轨——术语准确、数据具体、有出处。

## 答题手册（强制，2026-07-22 系统化定稿，取代旧版"回答深度"）

**第一性原则：答案的边界就是题目的边界。先判断题型，题型决定答案的结构和项目是否出场。长度不设下限也不设上限——题有大小，答有长短，07 题 150 字的快问快答和 01 题 300 字的机制展开都是定稿；真有东西可讲，一题写长完全可以。增删内容的唯一裁判是"这段能不能让读者学到东西、读得舒服"，不是字数。**

| 题型 | 识别特征 | 答法 | 项目出场 | 定稿参考 |
|------|----------|------|----------|----------|
| 实现题 | 怎么处理 / 怎么做 / 怎么发现 | 手段列表（带锚点数字）+ 1 个 #### why | 出场，报真实数值 | 01、10、15 |
| 原理题 | 为什么 | 理由列表，答完即收 | 不出场，数字可为零 | 05 |
| 概念题 | 是什么 / 区别 | 同层对比定位 + 代表性产品实例 + 真观点 | 可不出场 | 04 |
| 理解题 | 怎么看 / 有什么理解 | 一句一个定位，敢下"目前最强"判断，快问快答 | 不出场 | 07 |
| 开放题 | 未来 / 趋势 / 判断 | 判断 + 理由，给完即收 | 不出场 | 06 |

**通用禁令（所有题型，出现即删）**：

1. 金句不看题型看质量，裁判是遮句测试：遮住这句，读者少了信息或情绪，才留。烂金句 = 修辞复述——把刚讲完的机制换个修辞再说一遍（"会存不叫记忆，会取会丢才叫""每进来一条都要付租金""模型是引擎，harness 是整车"），遮住什么都不少，删。好金句 = 带增量——新判断（"就目前来说，Claude Code 就是最牛的终端 Agent"）、真幽默（"对，没错，定时任务"）、洞察（"每一个组件都是经过大量错误倒逼出来的"）、机制的精准概括（"谁离本次运行越近，谁说了算"），欢迎
2. 零收尾：不要总结段、不要升华句、不要跨章节叙事桥（"这就是后面那些题在问的事"），最后一个知识点说完直接结束，戛然而止是常态
3. 零小总起、零规则宣告："量级决定方案""再写死一条硬规则""重点来了"直接删，规则和事实自己会说话
4. 正面说完不补反面：同一件事只从一个方向讲
5. 不借题发挥：题目没问的（选型建议、行业优势、自家项目的取舍反思）不答；诚实交底全文至多 1 处
6. 零金句不等于零观点：对具体功能的批评性判断欢迎（"说实话，这个命令不该有"），敢喷具体的点

**why 名额**：一题选 1-2 个最值得展开的 why（通常升级成 ####），其余知识点只留规则不留解释。旁枝机制（辅助算法、估算方法）宁可不讲。

**数字纪律**：锚点数字按题型出场（实现题 2-4 个、原理题可为零）；同一个数字全文只出现一次，放在最需要它的论证位；估算类数值用约数（"差不多占到 80%"），面试时没人报小数点。

**语言细则**：

- 不缺字（最高优先级，见 SKILL.md 书面表达）：指代写全（"从 Agent 的善后工作中解放出来"）、名词短语写全（"推理的准确性""上下文窗口的大小"）、动词写全（"拆分"不写"拆"）
- 口语缓冲词是加分项："差不多""假如""而已""可能也就""就目前来说"
- 敢下最强判断，带时间限定："就目前来说，Claude Code 就是最牛的终端 Agent"
- 对比用读者熟悉的参照物（"记忆库一千多条，RAG 知识库几十个 G"），参照物一摆结论自现；比喻同样看质量不看数量——精准的旧知识锚点（crontab 之于后端读者）是好比喻，通用陈腐比喻（整车/引擎、高射炮打蚊子）和成链的拟物（闸门→过闸→出口）是烂比喻，删的是烂比喻，不是比喻本身
- 约定俗成的英文术语用原词（Prompt Caching、reward hacking）；自造的术语化短语（"优先级链"）换成平实动词短语（"按照优先级排列"）
- 热词的所指以当下业界实践为准：Harness Engineering、Loop Engineering 这类新词先查真实所指，用最有代表性的产品举例（Claude Code 的 /goal、/loop、Hooks、Sub-agent），不望文生义拿自家项目的旧概念硬套
- 同层对比定位法：定义新概念拉相邻概念并排划边界，一人一句话（提示词工程管"怎么说"、上下文工程管"知道什么"、Harness 管"能做什么不能做什么"）
- 模型和产品举例用当前代（Fable 5、GPT-5.6、GLM-5.2），禁止过时型号；口语自我确认（"对，没错，定时任务"）可用于预判读者的意外

## 推荐池导向硬规则（强制，2026-07 后台数据验证）

背景：推荐流生客平均停留 42 秒、完读率 8%～12%，去留和转化全部发生在前几屏。最佳模板是"技术Leader惊了"篇——推荐 4,674 之外拿到全场最高的关注转化（0.42%）、点赞率（1.23%）和收藏率（2.37%），靠的是下面四条：

1. **第一屏续演标题对话**。生客是被标题里的面试冲突勾进来的，正文第一屏必须把这场对话演下去并兑现悬念，再展开技术内容。禁止先铺新闻背景、先讲概念再入场。
2. **前 600 字完成一次完整交付**：对话演完 + 第一个硬核答案给完，让 42 秒就离开的读者也带走了东西。
3. **可收藏干货负载**：中段安排一个读者想存下来的单元（面试官追问清单、答案卡、技能对照表），它同时是老粉的可转发单元。干货负载旁边紧跟系列感关注钩子（如"AI 面试连环拷问第 N 篇，关注看全系列"），钩子必须出现在前 600 字到中段之间，不放文末。
4. **自家项目（PaiCLI 等）推广 ≤ 一屏，且不在最后一屏**；简历包装的核心职责最多 5 条，其余引导去 GitHub。同一项目不连续两篇推；连续篇目用轻推广样式——`## content` 开头放一个 PS 块：一句话定位（"类 Claude Code 的终端 Agent，已开源"）+ 仓库截图 + 引用块放 GitHub 链接，不展开简历包装。

**结尾写法（对标正例 ending 四拍）**：

1. 时代对比句，绑定本篇话题（"以前我们写 CRUD，现在我们需要懂协议、懂模型、懂 Agent、懂工程化"）
2. 加粗金句，给情绪价值，落在好奇心/成长视角
3. 展望升华一两句，允许带时代感（站在 AI 风口、参与技术变革），这是这个系列的固定情绪出口
4. 固定收尾："加油吧，兄弟姐妹们。"独立成段 + "下期见。"

## 技术细节尺度（面试类强制）

遵守 SKILL.md「技术术语尺度」全局规则（禁止自定义类名，保留标准库和业界术语）。面试类额外强调：

- 讲**原理和思路**，重点是**为什么这样设计**、**解决了什么问题**
- 面试时不可能记自定义类名，用通俗功能描述代替（"工具注册表""预算管控机制"）
- 回答要像真实面试中的表述，不是在念源码注释
