# AI 技术文章的生成工作流

## 环境声明

执行前跑 `date "+%Y年%m月%d日"` 拿当前日期。联网搜索关键词、frontmatter 的 date、正文时间描述，都用这个日期。

## 写作原则

### 交付标准（总纲，所有规则的最终裁判）

交付给读者的内容，必须经过认真思考和调研，确保读者能学到东西、没有理解歧义；在此基础上带适当的幽默风趣，让读者读起来舒服、喜欢、热血、认可。任何一条格式规则和这条冲突时，以这条为准。

### 语气和称呼

用"大家/我们/小伙伴"和读者拉近关系，少用"你"。

### 表达直抒胸臆（全文强制）

一句话能说清楚的事，不要用两句话绕弯弯。读者来看文章是为了拿到信息、学到知识、感受到情绪价值，不是听你做铺垫。

禁止的绕弯模式：先铺垫再说正事（"在介绍 X 之前，先了解一下 Y"）、假设式开场（"如果你遇到过 X，那么一定知道 Y"）、定义式起手（"所谓 X，就是指 Y"）、层层递进废话（"要理解 X，需要先明白 Y"）、同义反复（同一个判断换说法再说一遍）、无信息量的过渡（"说完了 A，接下来看看 B"）。

### 书面表达还是口语表达（全文强制）

判断方法：**这句话和技术相关吗？** 只要句子承载了技术信息（机制、原理、参数、数据、操作步骤、技术结论），一律走书面表达。

**书面表达**——所有技术相关的句子：

- **准确无误，清清楚楚**：用词准确无歧义，读者看一遍就懂。禁止用网络梗、模糊比喻、生僻词描述技术行为——读者看完觉得莫名其妙的词，一律换成书面表达
- **拟物比喻不成链**：自创的拟物说法（"过闸""闸门之后""出口装了限流"）最多出现一次，不许串成贯穿段落的比喻链；能用直白的功能词（限流、截断、检查）就不用比喻
- **不要缺字（最高优先级，历史最弱项，口语和书面都适用）**：主语、宾语、限定条件、指代对象写全，不指望读者靠语境脑补。每句话过一遍"小白测试"：假设读者第一次接触这个话题，句子里每个指代和省略——"循环""它""这个"——他知道指什么吗？不知道就写全。真实反例（均为定稿修正）：
  - "把人从循环里解放出来"→"把人从 Agent 的善后工作中解放出来"（"循环"是作者脑中的上下文，没写出来，读者莫名其妙）
  - "再往后排是记忆和协作"→"再往后是记忆和协作"（"排"是"排序"残留的半个词）
  - "拆给一队 Agent"→"拆分给多个 Agent"（动词写全，量词用准）
  - "钩到"→"够到"、"心态晃一下"→"说不羡慕是假的"（同音错字与生造词，用完整自然的词）
- **描述事实**：技术陈述句不要叠加修辞手法，判断和事实分开写——先陈述源码/文档怎么写的，再单独一句给自己的判断
- **可验证性**：每个技术断言要么有出处（源码、文档、实测），要么明确标注"我的推断是"，不把推断写成事实
- 禁止网络腔、拟人

**情感表达**——不承载技术信息，陈述的是我的情感、遭遇、感受：

- 允许真实的情感："有一说一，不黑不吹""太用心了兄弟"

### 前言表达（强制）

1. **打招呼**：「大家好，我是二哥呀。」独立成段
2. **钩子**：热点事件、反差冲突或直击痛点的问题（"有没有想过？Claude Code 的代码搜得又快又准，到底是怎么实现的？"）
3. **声明**：一句话说清楚为这篇文章付出了什么——"我花了一早上时间，翻了翻 Boris Cherny 的播客、亚马逊的论文、Claude Code 源码，把这件事从头到尾捋了一遍"。

**前言的姿态（全风格通用，比拍子更优先）**：

前言里的"我"是带队的人，不是评论员。评论员站在事件旁边分析给读者听，追求观点正确；带队的人站在读者身边指着前方说话，追求读者读完前言进入正文时的状态——被点名、被期待、想往下读。前言的 KPI 是这个状态，不是观点质量。三条落地：

1. **克制和大胆各归各位**：克制留给事实和观点（数字做减法、判断带"我觉得"、评价点到即止），大胆留给对读者的期许——敢承诺（"把握住这个机会，未来的七年，你也能……"），敢点名（"如果你是一位愿意相信努力的人"）。方向反了就是灾难：对事件下重结论 + 对读者不敢许愿 = 毒鸡汤评论员
2. **最后一句说给读者，收热不收冷**：恳切的邀请（"希望你能认真读一读"）永远胜过聪明的金句（"别人的故事看完就翻篇"）。前言最后一句是读者进正文前听到的最后一句话，要暖场，不要抖机灵
3. **写完自测**：数一数前言里说"他/它"的句子和说"大家/我们/你"的句子，前者明显多于后者，就是评论员姿态，重写

**爆料/薪资/他人成功类钩子的附加纪律（2026-07 定稿，强制）**：

用别人的高薪、offer、买房这类爆料做钩子时，前言必须积极正面，正视爆料、给足观点，再把读者引到行动上。定稿正例见 `docs/src/sidebar/itwanger/ai/agent-mianshi-tengxun.md` 的前言（腾讯 T10 篇，逐拍对照着写）。七拍结构：

1. **爆料转述，数字做减法**：正文段只保留一两个关键数（如"月薪 50k"），其余数字留给标题和爆料截图，不重复堆砌。爆点细节（全款买房）单独成段正视，不回避不删减。称呼用"这哥们"这类有温度的叫法，不用冷冰冰的"他/该员工"
2. **观点先认可**：点名爆料里真正有价值的部分，且要选反常识的那一个（"真正稀罕的不是 120w 的年包……是这哥们的定力"）。语气用"我觉得"，不下绝对断言
3. **再祛魅**：把神话拆回普通人路径——起点就是一场校招，每一步都是拿具体的交付换来的，"高薪不是一蹴而就，是七年慢慢积累下来的"。不造神，也不贩卖焦虑
4. **行业机会落到当下**：用行业事实收窄到本文方向（"大厂舍得给钱的岗位都是围绕 AI 展开的……尤其是 Agent，大厂小厂今年都在卷"），不喊口号
5. **正向承诺**：敢把爆料主角写成读者七年后的镜像——"如果能把握住这个机会，未来的七年，你也能像这哥们一样，在职场和生活上都稳稳地站住脚"
6. **信念筛选收束**：用"相信"排比把读者从旁观者点名成自己人——"如果你是一位愿意相信努力、相信过程、相信一步一个脚印的人，那接下来这份硬核的面经，希望你能认真读一读"。第 5、6 拍允许集中用"你"，计入全篇 ≤5 的预算
7. **引入正文**：面经来源用截图交代即可，调查投入声明在爆料类前言里可省略，投入感由正文的项目实答体现

另外两条底线：写作时评论和留言尚不存在，禁止出现"评论区""留言里"字样，读者情绪写成自己的第一反应，用完整自然的表达（"说不羡慕是假的"），不用"心态晃一下"这类生造说法；前言 450-700 字，七拍俱全，观点拍必须真展开，大胆下判断，不许只言片语带过。腾讯 T10 篇（agent-mianshi-tengxun.md）和两轮 15 问篇（agent-mianshi-tengxun-2.md）的前言均为定稿正例。

**两篇定稿对比出的细则（2026-07-22 增补）**：

1. **观点条数不硬凑**：真实想说几条就写几条，两条可以配"（欢迎大家补充）"收口；为了显全面硬凑第三条会露怯，human-tone 的三段式禁忌在前言同样适用。前言观点列表可用 ①、② 序号
2. **替读者发问要极短**："两年面六次，何必呢？"独立成段，不裹"很多人认为"的转述壳，质疑直接说出口再翻转
3. **观点讲完就走**：列表说完不再补对仗式总结金句（"果可以羡慕，因才值得学"这类收束句是说教尾巴，删）
4. **趋势判断给具体坐标**："去年主要是 RAG，今年的重心更多是 Agent"胜过"八股在降、AI 在涨"；判断敢用"毫无疑问"起头
5. **拍子是节拍不是模板**：信念排比、"下一个就是你"式大承诺，同款句式不许连续两篇复用，第二篇要么换新写法、要么压缩合并进机会拍
6. **对读者说话用口语直给**："面试挂掉不要否定自己"，不打磨成格言体（"把被拒当反馈，不当否定"）；格言味越浓，离带队人越远

### 正文结构

`## 01、标题` / `## 02、标题`。标题只写名称，不加冒号后缀解释，错误例子：`## 01、标题：解释`。三级标题 `### xxx`，不加分类前缀。

### 问题驱动（深度拆解类强制）

好的技术文章不是"我知道什么就写什么"，而是"读者会问什么，我按什么顺序回答"。

**操作方法**：
1. 确定主题后，先列出读者最可能问的 5-8 个问题（从"这是什么"到"怎么用好"逐步深入）
2. 砍掉太基础或太偏门的，保留 4-7 个串联起来
3. 每个章节标题就是一个问题（或问题的答案），章节内容就是回答这个问题
4. 问题之间可以有递进关系：前一个问题的答案自然引出下一个问题

每个问题的回答结构：**先给结论（一句话）→ 展开论证（证据+解读）→ 收一句实操建议**。问题链示例见 `references/deep-analysis.md`「结构模板」。

### 技术术语尺度（全文强制）

所有风格的文章都禁止出现项目自定义类名。读者不认识这些类名，堆砌只会让人困惑。

- **禁止**：项目内部的类名、方法名、变量名（如 `AgentBudget`、`McpServerManager`、`ToolInvocation`、`doExecuteTool`、`buildStepContext`）
- **保留**：Java 标准库（`ConcurrentHashMap`、`CompletableFuture`、`LinkedHashMap`）、业界公认术语（`JSON-RPC`、`MCP`、`Function Calling`、`ReAct`、`toolCalls`、`system prompt`）
- **替换方式**：用通俗的功能描述代替。`AgentBudget` → "循环预算机制"、`ToolRegistry` → "工具注册表"、`McpServerManager` → "MCP 管理模块"、`doExecuteTool` → "在执行工具时"

**英文术语首次标注**：英文术语首次出现时用"中文翻译（English term）"格式，如"异步生成器（async generator）"，后续直接用英文。MCP、RAG 等已约定俗成的缩写直接用，不展开全称。

### Case 创意

尽可能有趣，让读者眼前一亮。

涉及 Agent 可以和 PaiAgent 结合、RAG 可以和派聪明结合、CLI可以和 PaiCLI 结合（路径见步骤 2）。

### 段落与列表各得其所

正文默认用段落表达，用完整句子和自然过渡，长短句交替，符合公众号阅读习惯。**一个段落最多两个句号**——公众号在手机上读，段落一长读者就划走，到第二个句号就换段。能用一段话说清楚的事，不用列表。

**冒号纪律（对话体也不豁免）**：对话引导一律用句号（“老王点点头。”+ 独立引号段），禁止“老王问：”式冒号引导；叙述句里的冒号改成逗号、句号或拆句。只有三处允许用冒号：列表引导句（“做了三件事：”）、列表项的“术语：解释”格式、简历包装的字段名。

**但是，出现 3 个以上并列项时必须用列表**（第一/第二/第三、一是/二是/三是、方式一/方式二/方式三）。用"第一...第二...第三...第四..."的段落形式写多个并列项，读起来既累又有浓重的 AI 味。

判断顺序（从上往下，命中即停）：
1. 并列项能用"，"或"、"连成通顺的一句话（如"支持 A、B、C 三种格式"）→ 段落
2. 并列项 ≥ 3 个 → 列表
3. 并列项只有 2 个，且每项不超过一句话 → 段落

**一条列表项只说一件事**：不许用分号在一条里挤进第二件事（"按行分页……；超过 1MB 直接跳过"是两件事），第二件事要么另起一条，要么删。

## 特色元素

### 简历包装环节

文章涉及实战项目/GitHub 仓库时加一段：

项目名称：xxx
项目简介：xxx
技术栈：xxx、yyy
核心职责：

- （5 条，公式：用了什么技术栈，解决了什么问题、实现了什么业务、有哪些量化数据，不能出现自定义类名）

### 截图占位符（强制）

每个章节（包括二级标题、三级标题、四级标题）至少 1 个占位符——三级标题和它下面的四级标题各算一个章节，各自都要有。超过 500 字的章节安排 2 个占位符。

**位置规则（强制）**：占位符必须紧贴它可视化的内容——流程讲完紧跟流程图、列表讲完紧跟对应卡片、数据讲完紧跟数据板，放在读者刚读完文字、正需要一张图固化理解的位置。禁止习惯性堆在章节末尾；占位符和它对应的段落之间不能隔着其他话题的段落。

格式：

```
【截图：<名称>；风格：<风格>；截图目标：<证明什么>；关键词：<关键词1>、<关键词2>、<关键词3>】
```

**风格只能选以下 6 种之一**，不能自造：

| 风格 | 适用场景 |
|------|----------|
| `whiteboard` | 默认白板架构图，适合系统架构、模块关系、数据流 |
| `skill-card` | 技能卡片、目录树、少文字的结构展示 |
| `data-board` | 统计看板、柱状图、热力表、数据对比 |
| `three-layer` | 三层结构、金字塔、台阶、层级关系 |
| `swimlane` | 泳道流程、多人协作、并行时序 |
| `checklist-card` | 注意事项、准备清单、避坑卡片 |

示例：

```
【截图：Skill 的结构；风格：skill-card；截图目标：展示 Skill 目录里 SKILL.md 必需，references/scripts/assets 可选，并且按需加载；关键词：SKILL.md、references、scripts、assets、按需加载】
```

## 工作流程

### 步骤 1：读素材 + 选题质检

精读 `./sucai.md`，提取关键信息、数据、观点、截图。素材中的截图可以直接搬进正文，减少改稿成本。

**选题质检（读完素材后立即执行）**：

用 IKR 三维度评估素材是否撑得起一篇好文章：

- **I (Insight)** 有没有读者不知道的信息差？（源码发现、实测数据、反直觉事实、从未有人做过的横向对比）
- **K (Knowledge)** 读完能带走什么？（可执行的操作、可复用的认知框架）
- **R (Resonance)** 能戳中什么情绪？（"原来如此"的恍然、"我也遇到过"的共鸣、"竟然可以这样"的惊叹）

### 步骤 2：调查真实细节（强制）

项目相关内容必须先调查再写，不能凭通用知识猜。编出来的细节一眼假，浪费时间。

#### 角色边界

文章以二哥第一人称写作。

**AI 负责做的**（放心生成）：
- 补充技术背景知识（框架原理、API 文档、源码解读）
- 找证据和佐证（官方数据、GitHub 星标、基准测试）
- 按确定的角度扩写段落内容
- 梳理逻辑结构、调整章节顺序
- 调查公开信息（仓库、文档、博客）

#### 调查深度分层（按文章类型选择最低层次）

> 如果执行到此步骤时尚未确定风格，默认按中层调查。步骤 4 确定为「深度拆解类」后，必须回到本步骤补充深层调查。

| 层次 | 做什么 | 举例 | 适用风格 |
|------|--------|------|----------|
| 表层 | 搜官方博客、文档、公开榜单 | "官方说支持 X 功能" | 安装教程类（最低要求） |
| 中层 | 去 GitHub 看 README、issue、PR、commit 历史 | "这个 PR 的讨论里提到了设计取舍" | 产品评测类（最低要求） |
| 深层 | 读源码，找到具体实现，用代码片段证明观点 | "源码里这个常量是 0.01，意味着只占 1% context" | 深度拆解类（最低要求） |

深层调查的 5 步操作方法和代码片段筛选标准，见 `references/deep-analysis.md`「深层调查操作方法」，深度拆解类撰写前读。

**证据-解读交织写法（深度拆解类强制，其他类型鼓励）**：

不要只给结论。正确的节奏是：抛出问题 → 展示一手证据（源码/数据/官方原文）→ 用自己的话解读证据的含义。读者看到证据才会信服你的结论。正反示范见 `references/deep-analysis.md`「示范2：源码证据 + 解读」。

### 步骤 3：搜集资料 + 整理证据清单

补充可引用的公开信息（公开榜单、官方基准、第三方测试）。**准确数据必须访问原始来源**：GitHub 星标去仓库、榜单去 HuggingFace/LMSYS、跑分去官方发布。禁止二手转述，避免"听说""网友表示"。无法访问时注明"截至 YYYY-MM-DD"。

写正文前必须整理"引用证据清单"，至少包含：结论 + 来源链接 + 发布时间 + 为什么可以相信。未检索到证据时，清单里明确标记"未检索到有效证据"。禁止伪造数据。

### 步骤 4：文章风格选择

用 `AskUserQuestion` 让用户四选一：

- **安装教程类**（参考 `references/OpenClaw-install.md`）：手把手教学，注重实操指导
- **产品评测类**（参考 `references/deepseek-tui-review.md` 和 `references/deepseek-v4.md`，均为读者高赞验证过的正例）：有观点有数据，真实体验实况叙事
- **面试对话类**（参考 `references/anshui-yin-mianshi.md`，读者高赞验证过的正例）：要硬核，有技术支撑，对话体面试场景
- **深度拆解类**（写法指南 `references/deep-analysis.md` + 高赞正例全文 `references/claude-code-grep-vs-rag.md`）：读源码/官方博客，用一手证据拆解技术内幕，问题驱动结构

**选择判断**：如果素材的核心卖点是"好不好用、值不值得装"→ 产品评测类；如果核心卖点是"它内部怎么工作的、为什么这么设计"→ 深度拆解类。前者读者看完决定"装不装"，后者读者看完理解"为什么"。

**重要说明**：

1. **风格参考 ≠ 内容照搬**：参考对应文章学习语气、节奏、表达方式，内容必须大胆创新
2. **内容可以大胆**：基于你的理解和调查给真实场景、使用体验、case，不局限于 sucai.md
3. **开头结尾别老生常谈**：不要每次都套路化，根据内容特点设计有新意的开头结尾
4. **风格与素材不匹配时回退**：选定风格后发现素材深度不够支撑，退回步骤 1 重新评估 IKR，或请用户补充素材

### 步骤 5：撰写文章

**金句主动安排（强制）**：按 `human-tone.md`「金句用法」的触发时机和句式模板，撰写时主动铺设金句，一篇 5-10 处都可以，不设“宁缺毋滥”的心理上限。金句只进情感表达轨，硬禁区（证据、参数、推理链条附近）和遮句测试照旧生效——放开的是数量，不是位置。

文件格式 Markdown。正文目标 4400 字，初稿按 4400 字写，给自检删改留出 400 字余量，确保最终 ≥ 4000。

**字数的正确理解（2026-07-22 二哥原话校准）**：字数是最低交稿线，不是目标，更不是上限——内容真有价值，写到一万字也没问题。增删内容的唯一标准是读者能不能学到东西、读得舒不舒服，不是字数。题目数 ≥ 10 的面试对话文，答案长度由题型和内容价值决定（见 interview-style.md「答题手册」），全文最低线 3000 字，脚本报的"未达 4000"提示附说明通过。

1. 初稿完成跑 `./scripts/check_body_length.py` 检查
2. ≥ 4000 字：达标，进步骤 6
3. < 4000 字：不得交付。计算差额，**单次补充量必须 ≥ 差额 × 1.5**（例如差 800 字则一次至少补 1200 字），直接瞄 4300 以上，一轮补完。

落盘前必须跑完三层质检。**任何一层有未通过项，回到步骤 5 改稿，不得进入步骤 7**。自检完成后向用户展示报告。

#### L0 初学者重读（先于一切质检，强制）

以一个第一次接触这个话题的初学者身份，把文章从头到尾重读一遍，逐句问自己：这句话我能看懂吗？让初学者看懂、看明白，比一切都重要。重点抓四类：

- **生造表述**："产物可硬判"是生造，改成"产物能客观判定"才自然；任何自己压缩出来的词，换成日常会说的说法。判断标准不是"这个词存不存在"，是"读者学没学过"——冷门学科的术语（"词法匹配"出自编译原理，读者没学过）等同于生造，换成人人都懂的说法（"关键词匹配"）
- **术语无着陆**：行话第一次出现没有白话解释（"目标分布""带折扣的回报估计"），补一句"说白了"把它落地
- **指代悬空**："循环""它""这个"没有明确所指，读者对不上号
- **卡顿点**：读到哪里停了一下、需要回头重读，那里就是不通顺，改到一遍读顺为止

发现问题当场修，修完才进 L1。
