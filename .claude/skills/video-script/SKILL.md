# 短视频口播稿生成工作流

## 定位

面向二哥的短视频号，主打 3 分钟技术面试题拆解和 AI 技术科普。观众画像：准备面试的 Java/AI 开发者，刷短视频学知识点。

## 口播稿结构模板（3 分钟 ≈ 650-750 字）

经过实战验证的六段式结构，每一段都有明确的时间锚点和功能定位：

| 时间段 | 段落 | 功能 | 字数参考 |
|--------|------|------|----------|
| 0:00-0:15 | 钩子 | 抛错误答案 + 制造紧张感，让观众停下来 | 50-80 字 |
| 0:15-0:35 | 拆题 | 揭示面试官真正考什么（通常 3 个层次） | 80-120 字 |
| 0:35-2:15 | 满分回答 | 核心知识点，分步骤/分招数讲解 | 350-400 字 |
| 2:15-2:40 | 追问应对 | 面试官的深度追问 + 高水平回答 | 80-100 字 |
| 2:40-2:50 | 口诀 | 一句押韵的口诀，浓缩全文要点 | 30-40 字 |
| 2:50-3:00 | 收尾 | 引导关注 | 30 字 |

## 写作原则

### 节奏感

短视频的生命线是节奏。长短句交替，每 30 秒左右埋一个小钩子防划走。

- **短句制造冲击**："为什么？""注意。""重点来了。"
- **长句承载信息**：技术讲解用完整句，但一句话不超过 40 字
- **关键结论前留停顿**：用独立短段落断开，给观众反应时间

### 口语化但不随意

口播稿是念出来的，必须顺口。但技术描述要准确，不能为了顺口牺牲严谨。

- 用"怎么改？""靠什么？"这类设问句推进节奏
- 技术术语保留英文原文（Function Calling、JSON Schema、tool_calls、embedding、Top-K）
- 连接词用口语化的（"好，接下来""但如果""告诉他——"），不用书面连接词（"此外""与此同时""综上所述"）

### 信息密度

3 分钟只有 700 字左右的容量，每句话都必须有信息增量。

- 不说废话、不重复、不铺垫
- 一个知识点讲清楚就往下走，不用"换句话说""也就是说"复述
- 例子要具体到能在脑子里画面（"工具名 query-weather，参数 city 等于北京"）

### 画面感

观众在"看"视频，脚本要帮观众在脑子里建画面。

- 用具体场景代替抽象描述（"用户问了句北京天气怎么样"比"用户发起一次查询"有画面）
- 步骤讲解像在操作（"Agent 拿到 tool_calls，从注册表里找到对应的函数，直接调"）

## 去 AI 味（口播版）

口播稿的 AI 味比文章更致命，观众 1 秒就能感觉到不对。

**禁止出现**：
- 总结性套话："值得注意的是""需要指出的是""综上所述"
- 学术腔："本质上来说""从技术角度分析""我们可以发现"
- 互联网黑话："赋能""闭环""抓手""链路"
- AI 三段式：每个要点都走"概念→解释→例子"的固定模板
- 过渡废话："接下来让我们看看""话不多说""下面我来介绍一下"

**鼓励使用**：
- 直接的判断："那完蛋了""恭喜你，又踩坑了""这才是面试官想听的"
- 设问推进："靠什么？""怎么改？""怎么选？"
- 观众代入："如果你回答……""这时候你要自信地告诉面试官"

## 固定元素

以下元素在每期口播稿中保持一致，形成系列感：

- **开场**："哈喽大家好，我是二哥呀。"（放在钩子之后，不是最开头）
- **过渡到核心内容**："好，接下来给你满分回答，照着背就完事了。"（可微调措辞但保持"照着背"的语气）
- **追问过渡**："面试官如果追问/继续追问："
- **口诀引入**："最后一句口诀——"（可微调为"最后送大家一句口诀"）
- **收尾**："这道题你学废了吗？想解锁更多 Agent 面试题的源码级拆解，点赞关注，我是二哥，下期见！"

## 工作流程

### 步骤 1：确认主题和模式

两种模式：

1. **从零创作**：用户给一个面试题/技术主题，从零产出口播稿
2. **优化已有稿件**：用户给一份现有口播稿，进行节奏、表达、内容的优化

用 `AskUserQuestion` 确认不清楚的信息：主题的具体边界、目标时长（默认 3 分钟）、有没有必须覆盖的知识点。

### 步骤 2：深度调研（强制）

⚠️ **这一步不可跳过。** 口播稿虽然短，但技术点必须准确，出现原则性错误会直接翻车。

**必须启用子 Agent 做深度调研**，调研范围根据主题确定：

- 如果涉及某个技术机制（如 Function Calling、Skill 触发、RAG 检索）：调研其真实工作原理，读源码或官方文档
- 如果涉及某个产品/工具（如 Claude Code、Spring AI）：调研其最新特性和实际行为
- 如果涉及面试题：调研标准答案和常见误区

调研结果用于：
1. 确认口播稿中的技术描述是否准确
2. 找到能让内容更有深度的细节（比如"Skill 的 description 会被注入到 system prompt"这种内行才知道的细节）
3. 区分口播稿的主角是什么（比如 Skill ≠ Tool Call，不能混为一谈）

### 步骤 3：列大纲

按六段式结构列出大纲，明确：

- 钩子用什么错误答案制造冲突
- 面试官考察的 3 个层次是什么
- 核心内容分几步/几招，每步的一句话摘要
- 追问问什么，答案的亮点是什么
- 口诀怎么押

大纲列完先展示给用户确认，再进入正文撰写。

### 步骤 4：撰写口播稿

按大纲展开，注意：

- 总字数控制在 650-750 字（3 分钟）
- 每个段落之间空一行，方便看提词器
- 核心内容部分的每个步骤/招数独立成段
- 技术术语首次出现时用中文解释一次，后续直接用英文

写完后保存到 `docs/src/ai/video/` 目录，文件名用主题关键词，小写字母 + 连字符。

### 步骤 5：自检

落盘前跑一遍自检：

```
**口播稿自检** ✅/❌
- [ ] 字数：650-750 字区间 →（实际字数）
- [ ] 钩子：前 2 段有没有制造冲突/紧张感 →（用了什么手法）
- [ ] 技术准确性：核心知识点经过调研验证 →（调研了哪些来源）
- [ ] AI 味：无总结套话、无学术腔、无过渡废话 →（扫描结果）
- [ ] 节奏：长短句交替，无连续 3 句以上同结构 →（情况）
- [ ] 画面感：例子是否具体到能在脑子里成像 →（举例）
- [ ] 口诀：是否押韵或有节奏感，能脱口而出 →（口诀内容）
- [ ] 固定元素：开场/过渡/收尾是否齐全 →（检查结果）
```

自检通过后交付，未通过项回到步骤 4 修改。
